1 ML Fundamentals (they test this even for MLOps)

The bias–variance tradeoff

  • Bias = error from wrong assumptions → underfitting (model too simple). High train error.
  • Variance = sensitivity to training data → overfitting (model too complex). Low train error, high test error.
  • Goal: minimize total error = bias² + variance + irreducible noise.

Fixing overfitting

More data · regularization (L1/L2, dropout) · simpler model · early stopping · cross-validation · feature selection · data augmentation.

Fixing underfitting

More complex model · more/better features · reduce regularization · train longer.

L1 vs L2 regularization

L1 (Lasso)L2 (Ridge)
PenaltyΣ|w|Σw²
EffectDrives weights to exactly 0 → feature selection / sparsityShrinks weights smoothly, rarely to 0
Use whenMany irrelevant featuresCorrelated features, want stability

Elastic Net = L1 + L2 combined.

Supervised vs Unsupervised vs RL

  • Supervised: labeled data → classification / regression.
  • Unsupervised: no labels → clustering (K-means), dimensionality reduction (PCA), anomaly detection.
  • Reinforcement: agent learns via reward signal from environment.

Train / Validation / Test split

  • Train: fit parameters. Validation: tune hyperparameters / early stop. Test: final unbiased estimate, touched once.
  • K-fold cross-validation: split into k folds, train on k-1, validate on 1, rotate. Average the scores. Use when data is limited.
  • Stratified K-fold: preserves class ratios per fold — use for imbalanced classification.
  • Time-series split: never shuffle; train on past, validate on future (no leakage from the future).

Data leakage (a favourite trap question)

Leakage = information from outside the training set (or from the target/future) sneaks into features, giving falsely high offline scores that collapse in production.

Common causes: fitting the scaler/imputer on the full dataset before splitting; including a feature that's a proxy for the label; using future data in time series; target encoding without CV folds. Fix: fit all transforms inside a Pipeline on train only; split first, transform after.

Handling imbalanced data

  • Resampling: oversample minority (SMOTE), undersample majority.
  • Class weights: class_weight="balanced".
  • Metrics: never use accuracy — use precision/recall/F1, PR-AUC.
  • Threshold tuning instead of default 0.5.

Feature engineering basics

Scaling (StandardScaler/MinMax) · encoding (one-hot, label, target) · binning · interaction terms · handling missing values (impute mean/median/mode, or model-based) · log-transform skewed features · datetime decomposition.

Key algorithms — one-liners

AlgorithmUse / Idea
Linear / Logistic RegressionBaseline; interpretable; logistic = classification via sigmoid.
Decision TreeSplits on feature thresholds; interpretable but overfits.
Random ForestBagging of trees → reduces variance; robust default.
Gradient Boosting (XGBoost/LightGBM)Sequential trees correcting prior errors; SOTA on tabular.
SVMMax-margin separator; kernels for non-linear.
KNNLazy, distance-based; no training, slow inference.
K-MeansUnsupervised clustering into k centroids.
PCALinear dimensionality reduction via variance-max directions.
Neural NetsUniversal approximators; for unstructured data (image/text/audio).

Bagging vs Boosting: Bagging trains models in parallel on bootstrap samples to reduce variance (RF). Boosting trains sequentially, each correcting the last, to reduce bias (XGBoost).

2 Metrics & Evaluation

Confusion matrix terms

MetricFormulaWhen it matters
Accuracy(TP+TN)/AllBalanced classes only.
PrecisionTP/(TP+FP)Cost of false positive is high (spam, fraud flag → don't annoy good users).
Recall (Sensitivity)TP/(TP+FN)Cost of false negative is high (cancer, fraud detection → don't miss).
F12·P·R/(P+R)Harmonic mean; imbalanced data, single number.
ROC-AUCArea under TPR vs FPRRanking quality, threshold-independent. Misleading on heavy imbalance.
PR-AUCArea under Precision–RecallBetter than ROC-AUC for rare positives.
Log loss−Σ y·log(p)Penalizes confident wrong probabilities.

Regression metrics

  • MAE — average absolute error, robust to outliers.
  • MSE / RMSE — penalizes large errors; RMSE in target units.
  • R² — fraction of variance explained (1 = perfect, 0 = mean baseline).
  • MAPE — percentage error; breaks near zero values.
Interview move: when asked "which metric?", always answer with the business cost of each error type. e.g. "For fraud, a missed fraud (FN) costs far more than a false alarm (FP), so I optimize recall, then tune the threshold to cap precision loss."

Precision–Recall tradeoff

Lowering the classification threshold ↑ recall, ↓ precision (and vice-versa). Pick the threshold from the PR curve based on business constraints, not the default 0.5.

3 The MLOps Lifecycle

MLOps = DevOps principles (CI/CD, automation, monitoring, IaC) applied to ML, plus the extra dimensions of data and models.

End-to-end stages

  1. Problem framing — define target, success metric, constraints (latency, cost).
  2. Data ingestion & validation — pipelines, schema checks, quality gates.
  3. Data & feature versioning — DVC / feature store; reproducibility.
  4. Experimentation & training — track params/metrics/artifacts (MLflow/W&B).
  5. Model evaluation & validation — against baseline + business thresholds.
  6. Model registry — version, stage (Staging→Production), approvals.
  7. Packaging — containerize (Docker) with pinned deps.
  8. CI/CD — automated test → build → deploy on merge.
  9. Serving — REST/gRPC endpoint or batch job; autoscaling.
  10. Monitoring — latency, errors, data drift, model performance.
  11. Retraining loop — triggered by drift/decay/schedule → back to step 4.

How ML differs from traditional software

AspectTraditional SWML System
VersionedCodeCode + Data + Model + Hyperparameters
TestingUnit/integration+ Data validation, model validation, bias checks
DegradationStable unless code changesDecays silently as the world drifts
BehaviorDeterministicProbabilistic / data-dependent
CI/CDCI/CDCI/CD + CT (Continuous Training)

MLOps maturity levels (Google's model — good to cite)

  • Level 0: manual, script-driven, notebook-to-prod by hand. No automation.
  • Level 1: automated ML pipeline + continuous training; manual deploy of pipeline.
  • Level 2: full CI/CD for pipelines; rapid, reliable, automated retrain & deploy.

Reproducibility — how to guarantee it

Pin random seeds · version code (Git) · version data (DVC) · version environment (Docker + lockfile) · log all params/metrics (MLflow) · register the exact model artifact. "Given a run ID I can recreate the exact model."

6 Docker (must be fluent)

Why containerize a model

Reproducible env ("works on my machine" → "works everywhere"), dependency isolation, portability, easy scaling/rollback. Image = immutable snapshot; container = running instance.

Image vs Container vs Registry

  • Image — read-only template (layers).
  • Container — a running instance of an image.
  • Registry — store/share images (Docker Hub, ECR, GCR, ACR).

Sample ML Dockerfile (multi-stage)

FROM python:3.11-slim AS base
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY src/ ./src/
COPY model/ ./model/
EXPOSE 8000
CMD ["uvicorn", "src.app:app", "--host", "0.0.0.0", "--port", "8000"]

Key commands

docker build -t churn-api:1.0 .
docker run -p 8000:8000 churn-api:1.0
docker push myrepo/churn-api:1.0
docker ps        # running containers
docker logs  # logs
Optimization talking points: slim/alpine base images · multi-stage builds to drop build deps · .dockerignore · order layers so deps (rarely change) are cached before code · pin versions · run as non-root.

8 CI/CD (and CT) for ML

The three pipelines

  • CI (Continuous Integration): lint, unit tests, data validation, model validation (does new model beat baseline + thresholds?), build container.
  • CD (Continuous Delivery/Deployment): push image, deploy to staging → prod with approvals; canary/blue-green.
  • CT (Continuous Training): automated retrain triggered by schedule, new data, or drift → produces a candidate model that re-enters CI.

What's tested in ML CI that isn't in normal CI

Schema/data quality, feature distributions, training convergence, model metric ≥ threshold, no performance regression vs current prod model, fairness/bias checks, model size/latency budget.

GitHub Actions example (conceptual)

10 Monitoring, Drift & Retraining

What to monitor (3 layers)

  • Operational: latency, throughput/QPS, error rate, CPU/GPU/memory, uptime (Prometheus + Grafana).
  • Data: input schema, missing values, feature distributions vs training (drift).
  • Model: prediction distribution, confidence, and — when labels arrive — live accuracy/F1.

Drift types (know the difference)

TypeWhat changesExample
Data / Covariate driftP(X) — input distributionNew user demographics; sensor recalibrated.
Concept driftP(Y\|X) — input→output relationshipFraud patterns evolve; pandemic changes buying behavior.
Label / Prior driftP(Y) — target distributionFraud rate jumps from 1% to 5%.

Detecting drift

Statistical tests: PSI (Population Stability Index), KL divergence, KS test, Chi-square. Tools: Evidently AI, NannyML, WhyLabs, Arize, Fiddler.

PSI rule of thumb

<0.1 no significant change · 0.1–0.25 moderate, investigate · >0.25 major shift, likely retrain.

Retraining triggers

  • Scheduled (weekly/monthly) — simple, predictable.
  • Performance-based — metric drops below threshold (needs ground-truth labels).
  • Drift-based — PSI/KS crosses threshold.
  • Volume-based — N new labeled samples accumulated.
The hard part: getting ground-truth labels in production is delayed or expensive. So drift detection on inputs is often your early-warning system before true accuracy can be measured.

11 Feature Stores

A centralized layer to define, store, serve, and reuse features consistently for training and serving.

  • Offline store — historical features for training (warehouse/parquet).
  • Online store — low-latency features for real-time inference (Redis/DynamoDB).
  • Solves training–serving skew: same feature logic in both paths.
  • Tools: Feast (open-source), Tecton, SageMaker/Vertex/Databricks feature stores.
Training–serving skew = features computed differently (or with different data) at train vs serve time → silent accuracy loss. A feature store is the canonical fix.

13 LLMOps (bonus — increasingly asked in 2025/26)

  • RAG (Retrieval-Augmented Generation): embed docs → vector DB (FAISS/Pinecone/pgvector) → retrieve top-k → stuff into prompt. Reduces hallucination, no retraining for fresh knowledge.
  • Fine-tuning vs RAG vs prompting: prompt first (cheap), RAG for knowledge, fine-tune (LoRA/PEFT) for behavior/format/domain style.
  • Evaluation: harder — use LLM-as-judge, groundedness, faithfulness, human eval; track in tools like LangSmith.
  • Serving: vLLM / TGI for throughput; manage cost & latency; cache responses.
  • Guardrails: input/output filtering, prompt-injection defense, PII redaction.
If the JD mentions GenAI, have a one-line story: "I built a RAG pipeline over X docs using embeddings + a vector DB, served via FastAPI, and evaluated faithfulness with an LLM-judge."

def retry(n):
    def deco(fn):
        def wrap(*a, **k):
            for i in range(n):
                try: return fn(*a, **k)
                except Exception:
                    if i == n-1: raise
        return wrap
    return deco

15 ML System Design Framework

If asked "design an ML system for X," walk this skeleton out loud:

  1. Clarify: objective, scale (QPS, data size), latency SLA, online vs batch, success metric.
  2. Data: sources, labels (how obtained?), volume, pipeline, validation, storage.
  3. Features: engineering, feature store, avoid leakage/skew.
  4. Model: baseline → candidate; offline metric tied to the business KPI.
  5. Training infra: orchestration, experiment tracking, scheduled retrain.
  6. Serving: batch vs real-time, framework, autoscaling, caching.
  7. Deployment: registry, canary/shadow, rollback.
  8. Monitoring: ops + drift + performance; alerting; retraining trigger.
  9. Scale & cost: bottlenecks, GPU cost, batching, quantization.
Worked mini-example — recommendation system: implicit feedback data → candidate generation (collaborative filtering / two-tower embeddings) → ranking model (gradient boosting / DNN) → online serving with cached embeddings → A/B test on CTR → monitor CTR drift → retrain weekly.

16 Top 60 Q&A (expand & rehearse)

Say each answer aloud in 2–3 sentences. The dropdowns are model answers, not scripts to memorize verbatim.

A · MLOps core

1. What is MLOps and why do we need it?
MLOps applies DevOps practices (automation, CI/CD, monitoring, IaC) to the ML lifecycle, adding data and model versioning plus continuous training. We need it because ML models degrade as the world changes, are hard to reproduce, and getting a notebook model reliably into production — and keeping it healthy — requires automation and governance.
2. How does MLOps differ from DevOps?
DevOps versions code; MLOps also versions data, models, and hyperparameters. ML adds data/model validation to testing, models decay silently in production, and there's a third pipeline — Continuous Training (CT) — on top of CI/CD.
3. Walk me through an end-to-end ML pipeline.
Data ingestion & validation → feature engineering (with a feature store) → training with experiment tracking → evaluation against baseline + thresholds → register in model registry → package as a Docker image → CI/CD deploy via canary → serve as a REST endpoint → monitor ops + drift + performance → trigger retraining when drift/decay is detected.
4. What are the MLOps maturity levels?
Level 0: manual/notebook-driven. Level 1: automated training pipeline with continuous training. Level 2: full CI/CD for pipelines enabling rapid, reliable automated retrain & deploy.
5. How do you ensure reproducibility?
Version code in Git, data in DVC, environment in Docker with a lockfile; fix random seeds; log every param/metric/artifact in MLflow; register the exact model. Given a run ID I can recreate the identical model.

B · Data & training

6. What is data leakage and how do you prevent it?
Information from the target/future leaks into features, inflating offline scores that collapse live. Prevent by splitting first, fitting all transforms inside a Pipeline on train only, and never using future or label-derived features.
7. How do you handle imbalanced datasets?
SMOTE/oversampling or undersampling, class weights, threshold tuning, and the right metrics (precision/recall/F1, PR-AUC) — never accuracy.
8. Why version data, not just code?
The same code on different data yields a different model. To reproduce or debug a model you need the exact dataset that trained it — DVC ties a data snapshot to each Git commit.
9. What's training–serving skew?
Features computed differently at train vs serve time (different logic or data freshness), causing silent accuracy loss. A feature store with shared feature definitions fixes it.
10. How do you choose train/val/test for time series?
No shuffling — train on the past, validate/test on the future, with a forward-chaining (expanding window) split, to avoid leaking future information.

C · Deployment & serving

11. Batch vs real-time inference — when each?
Batch when predictions aren't needed instantly (nightly scoring) — cheap, high throughput. Real-time when each request needs an immediate prediction (fraud, recommendations) — REST/gRPC endpoint with autoscaling and a latency SLA.
12. How do you deploy a model with zero downtime?
Blue-green (switch traffic between two envs) or rolling updates in Kubernetes, often preceded by a canary on a small traffic slice to validate the new version before full rollout.
13. How do you roll back a bad model?
Models are versioned in the registry and shipped as immutable images, so I flip traffic back (blue-green) or run kubectl rollout undo. Canary/shadow usually catch the bad model before it reaches all users.
14. What is shadow deployment?
The new model receives real production traffic but its predictions aren't served to users — they're logged and compared against the live model, giving a risk-free real-world evaluation.
15. Canary vs A/B testing?
Canary is a release-safety technique — slowly ramp traffic to a new version watching system/model health. A/B testing is a statistical experiment to decide which model wins on a business KPI.
16. How would you containerize an ML model?
Write a Dockerfile from a slim Python base, install pinned deps, copy the model artifact and serving code (FastAPI), expose a port, and define the CMD. Use multi-stage builds and a .dockerignore to keep the image small, then push to a registry.
17. How do you reduce inference latency?
Model optimization (quantization, distillation, ONNX/TensorRT), request batching, caching frequent inputs, GPU serving (Triton), horizontal autoscaling, and feature pre-computation in an online store.

D · Monitoring

18. What do you monitor in production ML?
Three layers: operational (latency, errors, resource use), data (schema, distribution/drift), and model (prediction distribution and, when labels arrive, live accuracy).
19. Data drift vs concept drift?
Data/covariate drift = the input distribution P(X) changes. Concept drift = the relationship P(Y|X) changes even if inputs look the same. Concept drift is harder and usually demands retraining.
20. How do you detect drift?
Statistical tests comparing live vs training distributions — PSI, KS test, KL divergence, Chi-square — automated with tools like Evidently AI, with alert thresholds (e.g. PSI > 0.25).
21. When and how do you retrain?
Triggers: schedule, performance drop, drift threshold, or accumulated new labels. The retrain runs as an automated pipeline producing a candidate model that must pass the same validation gates before promotion.
22. Why is monitoring harder for ML than for normal services?
A service can be perfectly healthy (200s, low latency) while the model is quietly wrong. And ground-truth labels are often delayed, so you monitor input drift as an early-warning proxy for accuracy loss.

E · Tools & infra

23. What does MLflow do?
Experiment tracking (params/metrics/artifacts), a model packaging format, and a model registry with versioning and stages (Staging→Production) for governance.
24. What does DVC do?
Git-for-data: stores lightweight pointers in Git and the actual data in remote storage, and defines reproducible pipelines so you can recreate any model's exact data + code.
25. Airflow — key concepts?
DAGs of tasks defined in Python, run by a scheduler via an executor; operators define task types, XComs pass small data. Used for scheduled batch ETL/training orchestration.
26. Kubeflow vs Airflow?
Airflow is general-purpose orchestration; Kubeflow is ML-native and runs each step as a container on Kubernetes — better when you're already on K8s and need scalable, portable ML pipelines.
27. Docker image vs container?
An image is an immutable, layered template; a container is a running instance of that image. You build/push images and run containers.
28. Explain the core Kubernetes objects.
Pod (smallest unit), Deployment (manages replicas + rolling updates), Service (stable endpoint/load-balancing), Ingress (HTTP routing), ConfigMap/Secret (config), HPA (autoscaling).
29. What's a feature store and why use one?
A central layer to define, store, and serve features for both training (offline) and inference (online), eliminating training–serving skew and enabling feature reuse. Feast is the common open-source choice.
30. CI vs CD vs CT in ML?
CI: test code + validate data/model + build image. CD: deploy through staging→prod safely. CT: automated retraining triggered by data/drift/schedule that feeds new candidates back into CI.

F · Cloud

31. Walk me through deploying on SageMaker.
Store data in S3, run a managed training job (built-in or custom container), register the model, then deploy a real-time endpoint (or batch transform), with Model Monitor for drift and Pipelines orchestrating retrain.
32. How do you secure ML infra on the cloud?
Least-privilege IAM roles, secrets in a managed vault (not in code/images), VPC isolation, encryption at rest (KMS) and in transit (TLS), and image scanning in the registry.
33. Serverless vs container inference?
Serverless (Lambda/serverless endpoints) for spiky, low-volume traffic — scales to zero, no infra to manage but cold starts and size limits. Containers/K8s for steady high-volume, large models, or GPU needs.

G · ML fundamentals (quick fire)

34. Bias–variance tradeoff?
High bias = underfit (too simple), high variance = overfit (too complex). You balance model complexity to minimize total generalization error.
35. L1 vs L2 regularization?
L1 (Lasso) drives weights to exactly zero → feature selection/sparsity; L2 (Ridge) shrinks weights smoothly for stability with correlated features.
36. Precision vs recall — when which?
Optimize precision when false positives are costly (don't flag good transactions); optimize recall when false negatives are costly (don't miss fraud/disease). F1 balances both.
37. Why not accuracy for imbalanced data?
A 99%-negative dataset gets 99% accuracy by predicting all-negative while catching zero positives. Use precision/recall/F1 and PR-AUC instead.
38. Bagging vs boosting?
Bagging trains models in parallel on bootstrap samples to cut variance (Random Forest); boosting trains sequentially, each fixing the previous errors, to cut bias (XGBoost).
39. How does cross-validation help?
It gives a more reliable, lower-variance performance estimate by training/validating across multiple folds — important with limited data and for honest hyperparameter tuning.
40. ROC-AUC vs PR-AUC?
ROC-AUC measures ranking across all thresholds but can look optimistic on heavy imbalance; PR-AUC focuses on the positive class and is more informative when positives are rare.

H · Scenario / behavioral-technical

41. Your model is great offline but bad in production — what do you check?
Training–serving skew, data leakage in training, distribution drift between train and live data, feature pipeline bugs, label delay, and whether the offline metric actually reflects the business goal.
42. Accuracy suddenly dropped in prod overnight — debug it.
Check for an upstream data/schema change or pipeline failure first (most common), then sudden covariate drift, a bad deploy (roll back), or a broken feature source. Compare live feature distributions to training.
43. How do you decide between fine-tuning and RAG for an LLM use case?
RAG when the need is fresh/proprietary knowledge — no retraining, just update the index. Fine-tune when you need consistent behavior, format, or domain style. Start with prompting, add RAG, fine-tune last.
44. How would you reduce cloud ML costs?
Spot/preemptible instances for training, autoscaling and scale-to-zero for serving, right-sizing instances, batching requests, model optimization (quantization/distillation), and caching.
45. A stakeholder wants a model in 1 week — how do you respond?
Scope a strong baseline first (even a heuristic or simple model), define the success metric and data readiness, ship the baseline behind monitoring, then iterate — managing expectations on what's realistic.
46. How do you test a data pipeline?
Schema validation, null/range/uniqueness checks, row-count and freshness assertions (Great Expectations / custom), plus unit tests on transformation logic, run in CI.
47. How do you handle a feature that's frequently null in production?
Decide an imputation strategy consistent with training, add a missing-indicator feature, monitor its null rate, and alert if it spikes (often signals an upstream break).
48. Explain GitOps for ML.
Declarative infra/deploy config lives in Git as the single source of truth; a controller (Argo CD) continuously reconciles the cluster to match. Deployments become reviewable, auditable PRs with easy rollback.
49. How do you manage secrets in an ML pipeline?
Never in code or images — use a secrets manager (AWS Secrets Manager, Vault) or K8s Secrets, injected at runtime via env vars, with least-privilege access.
50. What's your model promotion process?
Candidate must beat the current prod model on holdout + business thresholds and pass bias/latency gates; it's tagged Staging, validated (shadow/canary), then promoted to Production in the registry with approval — fully traceable.

I · Rapid fire definitions

51. What is PSI?
Population Stability Index — quantifies distribution shift between two samples. <0.1 stable, 0.1–0.25 moderate, >0.25 significant drift.
52. What is the GIL?
Python's Global Interpreter Lock allows only one thread to execute bytecode at a time, so CPU-bound work doesn't parallelize with threads — use multiprocessing instead.
53. What is quantization?
Reducing model weight precision (e.g. FP32→INT8) to shrink size and speed up inference, with a small accuracy tradeoff.
54. What is a vector database?
A store optimized for similarity search over embeddings (ANN), used in RAG and recommendation — e.g. FAISS, Pinecone, pgvector.
55. What is IaC?
Infrastructure as Code — provisioning infra declaratively (Terraform/CloudFormation) so environments are versioned, reviewable, and reproducible.
56. What is a model registry?
A central, versioned store of trained models with lifecycle stages and metadata, enabling governance, promotion, and rollback.
57. What is an artifact in MLflow?
Any output file logged with a run — the serialized model, plots, configs — stored so the run is fully reproducible.
58. What is autoscaling (HPA)?
Kubernetes Horizontal Pod Autoscaler adds/removes pod replicas based on CPU/QPS/custom metrics to match load.
59. What is ONNX?
An open, framework-agnostic model format enabling portability and optimized runtime inference across hardware.
60. What is the difference between Continuous Delivery and Continuous Deployment?
Delivery = every change is release-ready and deployed with a manual approval gate; Deployment = changes go to production automatically with no manual step.

17½ HR Salary Negotiation (TCS)

TCS is band-driven, not free-market. You negotiate within a grade and escalate by justifying a higher grade — not by demanding a bigger number.

Reality check

  • Offers map to an internal grade. A 12+ yr lateral typically lands C3B / C4.
  • Lateral offers anchor on your current CTC + a "% hike" (commonly 20–40% on a switch).
  • Joining bonus, relocation, notice-period buyout are often more flexible than base.
  • Current CTC and offer details get verified at onboarding — never inflate.

Bangalore bands (lateral · directional, forum/aggregator-sourced)

Core scripts

  • "Current CTC?" — "Fixed ₹X, variable ₹Y, total ~₹Z. Moving into a more senior MLOps ownership role I'm looking at ₹A–B, but open to discussing the full package."
  • "Expectations?" — "Based on 12+ yrs, my AWS migration and LLMOps work, and market benchmarks, I'm targeting ₹[top of range]. Flexible on structure — base, joining bonus, or variable — if the total is fair."
  • Lowball / "band ceiling" — "Could we evaluate me for the next grade up? Multi-cloud K8s, ~50–80% cost optimization, end-to-end ownership maps to Consultant scope, not Associate." Then: "If base is capped, can we discuss a joining bonus or a 6-month review to bridge the gap?"
  • Notice period — "My notice is X days; I may be able to negotiate early release — does TCS support buyout?"

Direct account assignment + live phone call

Hired against a specific billable client role (not bench), the call may be unscheduled. Their urgency is your leverage.

  • Don't give away an early joining date for free — trade it for a better number.
  • Never get cornered into a snap number — "I'd rather give a considered number than a rushed one; can I call you back by end of day?" Costs nothing.
  • Keep your cheat sheet on the desk — current CTC, anchor ₹42–45 L, private floor. On a phone call you can read it.
  • Use silence — state your number, then stop talking.
  • Get it in writing before committing — "Can you send the formal offer with the fixed/variable breakup?" Don't verbally accept on the first call.
  • Confirm terms back before hanging up: "So — C4 band, ₹X fixed, ₹Y variable, joining by [date]?"
Don't: disclose your floor, give one hard number when asked first, or negotiate aggressively (TCS HR weighs flexibility & fit). Always: confirm the fixed vs. variable split — a high CTC with 25%+ variable is worth less than it looks.
